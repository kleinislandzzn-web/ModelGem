# Model Gem 模型效果评测与选型平台 Demo 交接文档

## 1. 项目背景

Model Gem 是一个面向 TikTok 内部跨团队协作的模型效果评测与选型 Demo。当前阶段重点不是做完整后端系统，而是先用一个可浏览、可演示、可继续迭代的网站，把图像模型和视频模型的评测资产、模型能力、合规状态、横向对比和业务选型逻辑串起来。

平台希望像“模型超市”一样，让设计、产品、算法、开发和业务同学可以快速理解：

- 现在有哪些图像/视频模型可用。
- 每个模型适合做什么，不适合做什么。
- 模型是否已经可接入业务，还是只能内部工具使用。
- 同一组 prompt / input 下，不同模型的效果差异是什么。
- 业务方输入需求后，应该优先参考哪些模型和历史考题结果。

## 2. 当前 Demo 范围

当前网站是前端 Demo，数据写在本地 React 代码里，没有真实后端和数据库。它适合用于：

- 内部方案沟通。
- 跨团队产品概念演示。
- 设计和信息架构评审。
- 后续和算法、平台、合规团队讨论 API / 数据结构 / 权限边界。

当前不包含：

- 真实模型 API 调用。
- 真实登录鉴权。
- 真实文件上传。
- 真实飞书文档解析。
- 后端持久化。

## 4. 页面结构

### 4.1 模型超市

模型超市是网站的第一个入口，核心概念是把所有可用模型像货架一样平铺展示。

当前分为两类：

- 图像模型：覆盖 T2I、I2I、局部编辑、风格迁移等任务。
- 视频模型：覆盖 T2V、I2V、视频编辑、镜头运动等任务。

每张模型卡片包含：

- 模型名称。
- 模型类型，例如 Checkpoint、Video Base。
- 支持任务。
- 合规状态。
- 媒体类型。
- 性能档位。
- 成本档位。
- 能力标签。
- 适合场景。
- 使用限制。
- 3:4 封面预览。视频模型当前使用 mp4 静音循环预览。

当前模型清单：

图像生成模型：

- Seedream 4.5：业务可用。
- Seedream 4.0：业务可用。
- Nano Banana V2：内部可用，不可直接接入业务上线。
- Nano Banana V1：内部可用，不可直接接入业务上线。
- Qwen Edit：业务可用。
- Seed Edit：业务可用。
- Seedream 5.0：待合规，未通过合规前不可业务使用。

视频生成模型：

- Seedance 2.0：业务可用。
- Seedance 2.0 Fast：业务可用。
- Seedance 2.0 Lite：待合规，待测批。
- Seedance 1.0 Mini：业务可用。

### 4.2 评测题库

评测题库用于展示已经沉淀下来的评测资产。这里的单位不是单条结果，而是一组题库入口。

每张题库卡片包含：

- 题库名称。
- 业务方向。
- 负责人。
- 题库内容类型，例如 Image、Video、Prompt。
- 任务类型。
- 媒体类型。
- 题目总数。
- 预览内容。
- 核心考点概览。

当前设计逻辑：

- 题库卡片不是详细结果页，而是入口。
- 点击题库后，可以跳转或联动到横向对比。
- “待整理题库”只在存在待整理题目时展示；如果待整理数量为 0，不显示该卡片。

### 4.3 横向对比

横向对比用于查看同一题库中多条 prompt 与多个模型输出的关系。

当前逻辑是图栈式表格：

- 左侧列展示 prompt 和 input 内容。
- 右侧横向罗列不同模型的输出结果。
- 每个结果格展示图片或视频结果、评分、结论和备注。
- 支持在页面内切换题库，不必返回评测题库页。
- 支持开启“考点高亮”，在 prompt 文本中用不同颜色标注关键考点。

设计决策：

- prompt 卡片不额外生成小标题，避免每次新增 prompt 都需要模型提炼标题，降低运行成本。
- Image / Video 类型不重复展示在每条 prompt 卡片内，因为题目组层面已经有媒体类型。
- 顶部弱化统计胶囊，重点放在题库选择、考点高亮和对比内容。

### 4.4 需求匹配

需求匹配面向业务方或设计师。用户可以输入一段需求，也可以粘贴 PRD、brief 或飞书文档链接。

当前界面包含：

- 需求文本输入框。
- 飞书文档 / 网页链接输入框。
- 系统解析出的重点考点。
- 建议的候选模型。
- 推荐理由。
- 风险提示。
- 参考历史题库结果。

当前匹配逻辑是前端规则模拟：

- 根据输入文本判断媒体类型。
- 根据关键词推断任务类型。
- 根据关键词匹配重点考点。
- 基于模型标签、历史结果和合规状态给出候选模型排序。

后续真实实现建议：

- 接入飞书文档解析。
- 将 PRD 内容结构化为媒体类型、任务类型、业务目标、素材条件、质量优先级、风险约束。
- 使用语义检索召回相似题库和历史结果。
- 推荐结果必须绑定证据，避免纯主观推荐。

### 4.5 一键试跑

一键试跑用于临时验证新 prompt。

当前用户流程：

1. 输入自己的 prompt。
2. 设置全局生成参数。
3. 多选模型，支持全选和清空。
4. 点击开始试跑。
5. 横向查看多个模型的临时输出结果。
6. 用户手动给每个结果打 1-5 星。
7. 如果觉得该 prompt 值得沉淀，可以点击“入库为考题”。

全局参数：

图像参数：

- 尺寸比例：1:1、3:4、4:3、9:16、16:9。
- 清晰度：1K、2K、4K。
- 默认：3:4、2K。

视频参数：

- 尺寸比例：1:1、3:4、4:3、9:16、16:9。
- 分辨率：480P、720P、1080P。
- 默认：9:16、720P。

设计决策：

- 这里不让用户直接选择已有考题，因为考题应该是已经验证并沉淀后的资产。
- 临时试跑结果默认不展示系统评分，分数由使用者自己打。
- 入库后的内容进入“待整理题库”，后续再由模型管理角色整理到正式题库。

### 4.6 接入指引

原“结果标注”页面已经调整为“接入指引”。

原因：

- 已经放入正式评测题库的结果，默认应当是人工校正和确认过的。
- 重新标注不应该作为主入口，否则会与评测题库的“已沉淀”属性冲突。
- 对业务和设计师而言，更有价值的是理解如何申请、接入、使用和复查模型。

当前页面包含：

- 接入流程说明。
- 不同模型接入模式。
- 合规状态。
- 负责人和使用说明占位。
- 接入前需要确认的事项。

## 5. 合规状态口径

当前状态分为三档：

- 业务可用：已经通过合规，可以接入业务并上线使用。
- 内部可用：只能在内部评测、内部工具、探索流程中使用，不可直接接入业务上线。
- 待合规：已确认推进或待测，但合规完成前不可业务使用。

注意：

- Nano Banana V1 / V2 当前是内部可用，不可直接接入业务。
- Seedream 5.0 当前待合规，未通过合规前不可业务使用。
- Seedance 2.0 Lite 当前待合规，属于待测批。

## 6. 权限角色

当前 Demo 保留两种角色概念：

- 模型管理：可以维护模型信息、整理题库、进行试跑和接入评估。
- 业务查看：主要用于查看模型、题库、横向对比和需求匹配结果。

当前 Demo 中切换角色不会隐藏功能，只会展示不同身份说明。后续真实系统中可以基于角色控制：

- 模型增删改。
- 合规状态修改。
- 题库入库和整理。
- 结果评分和结论修改。
- 模型 API 试跑权限。
- 业务方需求提交与查看权限。

## 7. 数据结构建议

后续如果接后端，建议至少包含以下对象：

- Model：模型信息、能力标签、合规状态、成本、性能、接入方式。
- BenchmarkSet：题库集合，包含业务场景、任务类型、考点、题目数量。
- BenchmarkCase：单条题目，包含 prompt、input、媒体类型、关联考点。
- TestResult：模型输出、参数、耗时、状态、评分、结论、备注。
- BusinessRequest：业务需求输入、解析结果、推荐模型、参考证据。
- DraftCase：一键试跑后尚未整理入正式题库的临时题目。

结果评分建议：

- 总分：1-5。
- 考点分：每个关联考点 1-5。
- 问题标签：主体崩坏、文字错误、风格漂移、运动不自然、时长不稳定、安全风险、背景穿帮等。
- 结论：推荐、谨慎推荐、不推荐、需复测。
- 备注：可以拆成面向测评团队和面向业务方两种口径。

## 10. 交接提醒

这个 Demo 当前最重要的价值是信息架构和跨团队沟通，不是技术完备性。接龙同学优先维护以下原则：

- 模型超市要清楚展示“能做什么”和“能不能业务上线”。
- 评测题库要像资产库，不要像临时结果列表。
- 横向对比要服务同题多模型对比，不要堆太多统计信息。
- 需求匹配必须给出证据，不要只给主观模型推荐。
- 一键试跑产生的好题要先进待整理，再进入正式题库。
